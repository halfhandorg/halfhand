Halfhand is a local-first CLI flight recorder for AI agents. It records agent
sessions — prompts, tool calls, MCP traffic, file changes, terminal output —
to a local SQLite database and replays them faithfully. **Halfhand makes zero
network calls.** Everything stays on your machine.

1.0.0 is the first stable release. From here the contracts in `STABILITY.md`
hold: CLI flags/exit codes, `--json` output, the on-disk database, MSRV, and
the `hh-core` Rust API change only additively (or through a documented
deprecation). A database written by `v0.1.0-beta.1` opens and reads correctly
under 1.0.

## Install

```bash
# cargo
cargo install halfhand

# Homebrew (after the tap is published)
brew tap halfhand-org/tap && brew install hh

# shell installer (Linux/macOS)
curl --proto '=https' --tlsv1.2 -LsSf https://github.com/halfhandorg/halfhand/releases/latest/download/hh-installer.sh | sh

# PowerShell (Windows)
irm https://github.com/halfhandorg/halfhand/releases/latest/download/hh-installer.ps1 | iex
```

Verify a binary's provenance with keyless Sigstore attestations:

```bash
gh attestation verify ./hh --repo halfhandorg/halfhand
```

## What's new

- **Secret redaction** — `hh scan` reports secrets without printing them and
  exits `4` on findings; `hh redact` irreversibly rewrites events and blobs in
  place, securely deletes originals, and compacts the DB.
- **Export & HTML replay** — `hh export --bundle` produces a portable `.hh`
  archive; `hh export --html` (and `hh replay --web`) produce a self-contained
  HTML replay page; `hh import` brings a bundle back under a new id. Exports
  are **always redacted by default**.
- **More adapters** — Codex CLI, Gemini CLI, and Claude Desktop, alongside
  Claude Code.
- **Search** — `hh search` is FTS5 full-text across all sessions, with filters
  and `--json`.
- **Windows** — builds on Windows every push (runtime PTY is best-effort per
  the SRS).
- **Stability contract** — `STABILITY.md` freezes the JSON schema at `2` and
  pins the CLI/DB/MSRV/`hh-core` rules.
- **Ergonomics** — `hh completions <shell>`, a `hh.1` man page, and a
  published mdbook docs site.

See `CHANGELOG.md` for the full list, and `docs/1.0-definition-of-done.md` for
the release bar with each item mapped to its proof.

## How we tried to earn the 1.0 bar

A few notes on the verification work behind 1.0 — the parts that don't show up
in a feature list.

### Every external-input parser has a fuzz target

Halfhand reads untrusted bytes: Claude/Codex/Gemini transcript JSONL, MCP
JSON-RPC stdio framing, import bundles, config TOML, redaction-detector input,
and zstd blob payloads. Each of those parsers has a `cargo-fuzz` target
(`fuzz/fuzz_targets/`) and a nightly CI job (`fuzz-nightly.yml`) that runs
them. The standing rule from `CLAUDE.md`: no `unwrap()`/`expect()` outside
tests and the top of `main()` — `?` with context everywhere — so a malformed
input is an actionable error, never a panic. We found and fixed
panic-on-truncation paths in the blob decompressor and the MCP frame splitter
during fuzzing before they ever reached a user.

### Storage invariants are property-tested, not example-tested

The things that would be catastrophic if wrong — blob refcounts going negative
or leaking, migrations applied twice diverging from applied once, step-ordinal
assignment, and export→import→export stability — are `proptest` suites, not
hand-written examples:

- `blob_refcount_prop` and `prop_redact.rs`'s `assert_refcounts_balanced`
  prove the refcount books balance after `delete_session` and after a redact
  rewrite (every live reference has a blob; no orphaned blob has a live
  reference). This is the test that lets us claim "redaction securely deletes
  the original" with a straight face.
- `migration_idempotency_prop` proves applying the migration set twice yields
  the same DB as applying it once — the property an upgrading user relies on
  when their `1.x` opens a `1.(x-1)` database.
- `assign_steps_invariants` (proptest) + `assign_steps_idempotent` prove the
  step-ordinal assignment is deterministic and re-runnable.
- `export_import_export_round_trips_content` proves a session survives an
  export→import→export cycle content-stable (the round-trip the `hh import`
  contract is built on).

### Migration from the wild

`hh/tests/migration_from_beta.rs` opens a database **actually produced by
`v0.1.0-beta.1`** (the committed fixture at `hh/tests/fixtures/beta-db/`,
generated by checking out the beta tag and running the fixture agent) with
the 1.0 binary, and asserts list / inspect (valid JSON) / export (JSON +
bundle) / replay (`--web` HTML) all succeed. It runs in CI on every push, so a
regression that breaks forward-readability of an old database fails the build
immediately.

### The audit pass

Before tagging we ran the full gate: `cargo clippy --workspace --all-targets
-- -D warnings` (pedantic, deny warnings), `cargo fmt --check`, rustdoc with
`-D warnings` (broken intra-doc links are now a hard error — we cleaned a
backlog of stale module links and bare URLs that had been silently broken),
`cargo deny check`, `cargo audit` (three "unmaintained/unsound" transitive
warnings acknowledged with justifications in `audit.toml`/`deny.toml` — none
exploitable, none in code we call), `cargo semver-checks` for `hh-core` (196
checks pass), and the criterion bench regression gate (nightly, fails on >15%
median-time regression).

## Known limitations (unchanged from the beta, now documented as 1.0 scope)

- **Windows PTY is best-effort** (SRS §2.2): Windows builds and installs, but
  PTY recording there is not validated to the same bar as Linux/macOS.
- **Replay is faithful playback, not deterministic re-execution**: the agent
  is not re-invoked during replay; no network calls are made. A session whose
  recording was interrupted replays the partial timeline and is marked
  `interrupted`.
- **Direct `hh.db` SQL is a debugging interface**, not the contract. The
  supported programmatic interfaces are `--json` and `hh export` bundles.

## Upgrading from v0.1.0-beta.1

Just install 1.0.0. Your existing data dir opens as-is; no migration step is
needed (the schema was frozen at `2` at the beta.1 cut). If you pinned the
beta in CI, bump the pin to `1.0.0`.
