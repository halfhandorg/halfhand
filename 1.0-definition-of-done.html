Halfhand

Halfhand 1.0 — definition of done

This page walks the v1.0.0 release bar from CLAUDE.md's "v1.0.0 addendum" (six bullets) plus STABILITY.md §a–e, and for each item points at the test, doc, or artifact that proves it (or notes the gap). It is the checklist used to sign off the 1.0.0 tag. Nothing here is a new contract — it maps existing contracts to existing proofs.

The hard gate before tagging is the hand-run docs/manual-qa.md checklist. This page is the automated/auditable half: everything you can point a CI run or a file at.

CLAUDE.md v1.0.0 addendum — six bullets

1. Backward compatibility is a feature

Any change to CLI flags, --json output, the DB schema, exit codes, or env vars is additive or goes through the STABILITY.md deprecation policy. A breaking change stops and is proposed before implementing.

  • Policy doc: STABILITY.md §a–e (CLI, --json, on-disk DB, MSRV, hh-core semver). Shipped in eff307d ("Add STABILITY.md, freeze JSON schema at 2").
  • Forward-readability proof (DB): hh/tests/migration_from_beta.rs opens a database created by v0.1.0-beta.1 with the 1.0 binary and asserts list / inspect / export (JSON + bundle) / replay (--web HTML) all succeed. The beta DB is a committed fixture at hh/tests/fixtures/beta-db/ with a provenance README.md.
  • API-stability proof (hh-core): cargo semver-checks runs in CI (.github/workflows/ci.yml, semver-checks job) against origin/main with --release-type minor and --exclude enum_marked_non_exhaustive (the one-time cleanup allowance recorded in STABILITY.md §e).
  • Deprecation machinery: hh-core/src/deprecation.rs exists for the "warns for one minor release, then removes" cycle (§a). No 1.0 feature uses it — 1.0 is additive — but the rail is in place.

2. Every parser of untrusted input has a fuzz target and never panics

Parsers of Claude/Codex/Gemini JSONL, MCP JSON-RPC framing, import bundles, and config files must have a fuzz target and must never panic on malformed input — errors only.

  • Fuzz targets (fuzz/fuzz_targets/):
    • claude_jsonl.rs — Claude Code transcript JSONL (FR-1.5 adapter).
    • codex_jsonl.rs — Codex CLI JSONL adapter.
    • gemini_jsonl.rs — Gemini CLI JSONL adapter.
    • mcp_frame.rs — MCP JSON-RPC stdio framing (FR-2 proxy).
    • import_bundle.rs — hh import bundle manifest + payload.
    • config_toml.rs — [redaction] / config TOML.
    • redact_detect.rs — secret detector on adversarial input.
    • blob_decompress.rs — zstd blob decode (corrupt/truncated blobs).
  • CI: .github/workflows/fuzz-nightly.yml runs the targets on a schedule.
  • No-panic discipline: CLAUDE.md forbids unwrap()/expect() outside #[cfg(test)] and the top of main(); ? with context everywhere. The audit pass (bullet 2 of the release tasks) confirmed no stray dbg!/println! in library code and clippy pedantic clean.

3. Storage invariants are property-tested, not example-tested

Blob refcounts, migration idempotency, step-number assignment, and export→import round-trips are property tests.

  • Blob refcounts: hh-core/src/store.rs mod blob_refcount_prop (inline proptest) + hh-core/tests/prop_redact.rs assert_refcounts_balanced (refcount books balance after redaction rewrites; referenced blobs exist).
  • Migration idempotency: hh-core/src/store.rs mod migration_idempotency_prop (proptest) and migration_is_idempotent (unit); runner doc at hh-core/src/migrations.rs (DR-1, forward-only, idempotent).
  • Step-number assignment: hh-core/tests/prop_step.rs assign_steps_invariants (proptest) + hh-core/src/step.rs assign_steps_idempotent.
  • Export→import round-trip: hh-core/tests/prop_bundle.rs export_import_export_round_trips_content (export → import → export is content-stable). Also covered end-to-end by hh/tests/migration_from_beta.rs's bundle export + the import tests in hh/tests/cli.rs.

4. Performance is regression-gated

Criterion benches live in benches/, and CI fails on >15% regression vs the committed baseline.

  • Benches: hh-core/benches/storage.rs, adapter_jsonl.rs, search.rs (criterion, declared in hh-core/Cargo.toml).
  • Regression gate: .github/workflows/bench-nightly.yml runs cargo bench -p hh-core --features fuzzing --locked nightly and calls .github/scripts/bench_compare.py to fail on >15% median-time regression vs the cached baseline. Nightly (not per-PR) per the addendum's allowance.
  • Local repro: just bench runs the same benches.

In the same PR, or the feature is not done.

6. Exit codes are part of the contract

0 ok, 1 generic error, 2 usage error, 3 session-not-found, 4 redaction/policy block — documented and tested.

  • Doc: STABILITY.md §a, the exit-code table (codes 0/1/2/3/4 with meanings).
  • Tests: hh/tests/cli_conformance.rs — usage_error_exits_2, unknown_session_exits_3, scan_with_findings_exits_4 (and --help/sub --help exit 0). The table is the single source of truth; tests pin it.

STABILITY.md §a–e

(a) CLI: flags, subcommands, exit codes

Additive freely; removal/behavior-change requires a deprecation cycle via hh-core/src/deprecation.rs. Exit-code table tested in cli_conformance.rs (above). --help output for every subcommand is snapshot-tested, so a wording/flag change is a visible diff that forces review.

(b) --json

Additive only (new fields/body shapes/enum values); a breaking change ships as a new schema value with both shapes documented side-by-side in docs/json.md, gated behind a major version. JSON schema frozen at version 2 (eff307d); docs/json.md documents the shape. --json output is exercised by hh/tests/cli.rs and hh/tests/migration_from_beta.rs (list --json parses; inspect --json is valid NDJSON).

(c) On-disk database

Any 1.x release can open and read a DB written by any earlier 1.x release; migrations only add. Proven by hh/tests/migration_from_beta.rs (beta.1 → 1.0 forward readability) and migration_idempotency_prop (re-applying migrations is a no-op). hh.db is a documented interface (DR-2); the schema is in hh-core/src/migrations/.

(d) MSRV

MSRV is 1.75 (CLAUDE.md); an MSRV bump only happens in a minor release. Enforced by ci.yml's test matrix pinning the MSRV toolchain.

(e) hh-core semver policy

hh-core follows semver; breaking changes bump the major. Non-exhaustive enums (AgentKind, …) keep new variants additive. Enforced by the semver-checks CI job (bullet 1). The one-time enum_marked_non_exhaustive exclusion is documented in STABILITY.md §e and passed on the 1.0 cut (196 checks pass).

Gaps found while walking this list

None open. Every addendum bullet and every STABILITY clause maps to a committed proof above. The one item that is not a per-PR CI gate — the criterion regression check — is intentionally a nightly job per the addendum's explicit allowance (bench-nightly.yml).

The remaining sign-off is the human half: docs/manual-qa.md, run by hand on Linux, macOS, and Windows with real Claude Code / Codex / MCP proxy before tagging. Do not tag until that checklist passes.
